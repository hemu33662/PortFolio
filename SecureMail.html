Project Overview

A centralized gateway for enterprise-grade email management.

The Challenge

Managing SMTP configurations across multiple applications is repetitive and insecure. Direct exposure of SMTP credentials in client-side or distributed microservices increases the risk of data breaches.

Secure Mail Service solves this by acting as a single, hardened entry point for all organizational mail requests, abstracting configuration complexity while enforcing strict security protocols.

  • Multi-Tenant Client Registry
  • HMAC Signature Verification
  • Replay Attack Mitigation
  • Multipart Attachment Support

Security Architecture

Implementing Zero-Trust Authentication via HMAC-SHA256.

bash - interaction.sh
# Authenticated Request Example
curl -X POST http://localhost:8080/api/v1/mail \
-H "X-API-KEY: YOUR_KEY" \
-H "X-USER-ID: dev_test" \
-H "X-SIGNATURE: BASE64_HMAC(...)" \
-H "X-TIMESTAMP: 1712056000" \
-F "to=recipient@example.com" \
-F "subject=Safety Report" \
-F "body=<h1>System Alert</h1>"
HMAC Signing

Requests are signed client-side using a secret key that is never transmitted over the network.

Timestamp Validation

Strict 5-minute window validation prevents malicious actors from capturing and replaying requests.

Multi-Tenant Isolation

Each client has localized authorization, preventing sender spoofing and cross-tenant access.

Technical Foundation

Built with modern Java technologies and enterprise patterns.

Core Implementation

The service is built on Spring Boot 3.x, leveraging the power of Jakarta Mail for robust SMTP interfacing. The heart of the security system is a custom ApiKeyAuthFilter which performs real-time HMAC validation before requests reach the controller.

  • Custom Authentication Filter A filter-based security implementation that regenerates HMAC signatures server-side to verify request integrity.
  • JSON Registry System A dynamic clients.json registry allowing for hot-swapping client credentials and authorized sender lists.
  • Structured Audit Logging Using Logback and SLF4J, every request is tracked via a traceId for enterprise auditability and debugging.
